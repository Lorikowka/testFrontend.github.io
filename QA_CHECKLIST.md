# Документация для проверки сайта

Этот чек-лист нужен, чтобы вручную и технически проверить сайт записи и оплаты перед публикацией или после изменений.

## 1. Подготовка

Проверку лучше проводить в отдельной ветке и в тестовом режиме оплаты.

Проверьте, что установлены:

- Node.js 18 или новее.
- npm.
- Python 3.10 или новее, если проверяется Telegram-бот.

Перед стартом убедитесь, что в `backend/.env` настроены минимум:

```env
PORT=1488
MOCK_MODE=true
SITE_URL=http://localhost:1488
API_KEY=your_secret_api_key
ADMIN_API_KEYS=your_admin_api_key
BOT_API_KEYS=your_bot_api_key
```

Для проверки реальной YooKassa вместо `MOCK_MODE=true` используйте тестовые ключи YooKassa и `MOCK_MODE=false`.

## 2. Быстрые технические проверки

Из корня проекта:

```powershell
node --check backend\server.js
node --check backend\database.js
node --check backend\migrations.js
node --check backend\lib\security.js
node --check backend\lib\sessionBooking.js
node --check frontend\js\main.js
python -m py_compile telegram-bot\bot.py
```

Из папки `backend`:

```powershell
npm test
npm run security:check
```

Успешный результат:

- Все `node --check` завершаются без вывода ошибок.
- `python -m py_compile` завершается без ошибок.
- `npm test` показывает успешные проверки.
- `npm run security:check` не показывает критических ошибок. Предупреждение о наличии локального `.env` допустимо, если файл игнорируется git.

## 3. Запуск сайта локально

Из папки `backend`:

```powershell
npm install
npm start
```

Откройте:

```text
http://localhost:1488
```

Проверьте health endpoint:

```text
http://localhost:1488/api/health
```

Ожидается JSON со статусом `ok`.

## 4. Проверка главной страницы

Проверьте в браузере:

- Страница открывается без ошибок в консоли.
- Логотип, фото, дипломы и стили загружаются.
- Меню работает на desktop и mobile.
- Кнопка записи прокручивает к форме.
- Все основные блоки видны: о специалисте, услуги, дипломы, запись, контакты.
- Текст не налезает на соседние элементы на мобильной ширине.

Минимальные размеры для проверки:

- Desktop: 1440x900.
- Tablet: 768x1024.
- Mobile: 390x844.

## 5. Проверка формы записи

Сценарий успешного выбора:

1. Откройте блок записи.
2. Выберите услугу.
3. Выберите доступную дату.
4. Выберите время.
5. Заполните ФИО и телефон.
6. При необходимости заполните email и комментарий.
7. Проверьте итоговый блок перед оплатой.
8. Нажмите кнопку оплаты.

Ожидается:

- Без ФИО и телефона форма не отправляется.
- Без даты и времени форма не отправляется.
- Цена соответствует выбранной услуге.
- Доступные слоты отображаются корректно.
- Занятый слот после обновления расписания недоступен.

## 6. Проверка оплаты в MOCK_MODE

При `MOCK_MODE=true` оплата должна проходить без перехода в YooKassa.

Сценарий:

1. Заполните форму записи.
2. Нажмите оплату.
3. Дождитесь перехода на `payment-check.html`.
4. Дождитесь возврата на главную страницу.

Ожидается:

- Появляется сообщение об успешной оплате.
- В базе создается запись о платеже.
- В базе создается запись о сеансе.
- Повторный переход по ссылке проверки не создает дубль записи.

## 7. Проверка реальной тестовой YooKassa

Используйте только тестовый магазин и тестовые ключи.

В `backend/.env`:

```env
MOCK_MODE=false
YOOKASSA_SHOP_ID=...
YOOKASSA_SECRET_KEY=...
YOOKASSA_WEBHOOK_SECRET=...
SITE_URL=https://ваш-тестовый-домен
```

Сценарии:

- Успешная оплата тестовой картой.
- Отмена оплаты на стороне YooKassa.
- Возврат на `payment-check.html`.
- Попадание на `payment-failed.html` при ошибке или отмене.

Проверьте:

- Статус платежа меняется на `succeeded` после успешной оплаты.
- При отмене статус становится `canceled` или `expired`.
- Webhook YooKassa принимается только с валидной подписью.
- Сеанс создается только после успешной оплаты.

## 8. Проверка защиты от дублей и вместимости

Индивидуальная консультация:

1. Создайте запись на конкретную дату и время.
2. Попробуйте создать вторую индивидуальную запись на тот же слот.

Ожидается:

- Второй клиент получает ошибку занятости слота.
- В базе не появляется второй активный сеанс на то же время.

Групповая услуга:

1. Выберите групповую услугу.
2. Создайте несколько записей на один слот разными клиентами.
3. Попробуйте повторно записать того же клиента по телефону или email.
4. Доведите количество участников до `capacity`.

Ожидается:

- Разные клиенты могут занять места до лимита.
- Один и тот же клиент не может записаться повторно.
- После достижения лимита слот становится недоступным.

## 9. Проверка admin API

Подставьте свой ключ:

```powershell
$headers = @{ "X-API-Key" = "your_admin_api_key" }
Invoke-RestMethod -Uri "http://localhost:1488/api/sessions?limit=10" -Headers $headers
Invoke-RestMethod -Uri "http://localhost:1488/api/payments?limit=10" -Headers $headers
```

Проверьте без ключа:

```powershell
Invoke-WebRequest -Uri "http://localhost:1488/api/sessions?limit=10"
```

Ожидается:

- С ключом возвращается список.
- Без ключа возвращается `401`.
- `X-Bot-API-Key` работает только если ключ есть в `BOT_API_KEYS`.

Проверка смены статуса:

```powershell
$headers = @{ "X-API-Key" = "your_admin_api_key"; "Content-Type" = "application/json" }
$body = @{ status = "completed" } | ConvertTo-Json
Invoke-RestMethod -Method Patch -Uri "http://localhost:1488/api/sessions/1/status" -Headers $headers -Body $body
```

Ожидается:

- Статус меняется на `completed`.
- Неверный статус отклоняется.

## 10. Проверка Telegram-бота

В `telegram-bot/.env`:

```env
TELEGRAM_BOT_TOKEN=...
TELEGRAM_ADMIN_ID=...
BACKEND_URL=http://localhost:1488
API_KEY=your_bot_api_key
```

Запуск:

```powershell
cd telegram-bot
pip install -r requirements.txt
python bot.py
```

Проверьте команды и кнопки:

- Сегодня.
- Завтра.
- Эта неделя.
- Все записи.
- Платежи.
- Детали записи.
- Отменить запись.
- Отметить как проведенную.
- Экспорт календаря.

Ожидается:

- Бот отвечает только администратору.
- Данные приходят из backend API.
- Кнопка `Проведен` меняет статус через API.
- Кнопка отмены отменяет запись.

## 11. Проверка безопасности frontend

В браузере откройте DevTools:

- Console: нет JavaScript-ошибок.
- Network: HTML, CSS, JS, изображения загружаются со статусом 200.
- Application: черновик оплаты хранится в `sessionStorage`, а не в `localStorage`.
- Security headers есть при отдаче через backend.

Проверьте, что чувствительные данные не появляются:

- В URL, кроме одноразовых payment token параметров.
- В console logs на production.
- В git status как `.env` или база данных.

## 12. Проверка базы данных

База находится в:

```text
backend/data/payments.db
```

Проверить последние записи можно временным Node-скриптом:

```powershell
cd backend
node -e "const sqlite3=require('sqlite3').verbose(); const db=new sqlite3.Database('data/payments.db'); db.all('select id,status,service_name,session_datetime,client_name from sessions order by created_at desc limit 10', [], (e,r)=>{console.log(e||r); db.close();});"
```

Ожидается:

- Нет дублей индивидуальных активных слотов.
- У групповых записей количество участников не превышает `capacity`.
- Отмененные записи имеют статус `cancelled`.
- Проведенные записи имеют статус `completed`.

## 13. Перед публикацией

Обязательный финальный список:

- [ ] `npm test` проходит.
- [ ] `npm run security:check` без критических ошибок.
- [ ] `.env` не попадает в git.
- [ ] `backend/data/*.db` не попадает в git.
- [ ] `MOCK_MODE=false` только если подключена тестовая или реальная YooKassa.
- [ ] `SITE_URL` соответствует реальному домену.
- [ ] `ALLOWED_ORIGINS` содержит только нужные домены.
- [ ] `ADMIN_API_KEYS` и `BOT_API_KEYS` заданы и отличаются для продакшена.
- [ ] Тестовая успешная оплата пройдена.
- [ ] Тестовая отмена оплаты пройдена.
- [ ] Telegram-бот видит записи и умеет менять статус.

## 14. Что приложить к отчету о проверке

После проверки сохраните:

- Дата и время проверки.
- Ветка/коммит.
- Режим оплаты: `MOCK_MODE=true` или YooKassa test.
- Список пройденных сценариев.
- Скриншоты главной страницы, формы, успешной оплаты и ошибки оплаты.
- Вывод команд `npm test` и `npm run security:check`.
- Список найденных проблем с шагами воспроизведения.
